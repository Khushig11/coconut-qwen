Coconut Qwen Report

This report explains what I implemented for the Coconut LR-2 task, how I understand continuous latent reasoning, the problems I faced while implementing it, and the evidence from my single-example GSM8K overfit experiment.

1. What I did

I implemented a small Coconut-style reasoning experiment using a Qwen3 model. The required bottom-line goal was to show that the model can overfit one GSM8K example when part of the normal chain-of-thought reasoning is replaced with continuous latent thoughts. 

python final_train.py

The experiment uses one GSM8K math problem, inserts the Coconut special markers, runs latent reasoning steps, trains on the single example, saves the loss curve, and saves the generated answer in outputs/overfit_results.json.

Important settings from my final run:

  • Base model: Qwen/Qwen3-0.6B
  • Dataset example: one GSM8K question
  • Latent thoughts: k_latents = 2

2. How Coconut works in my own words

A normal language model reasons by generating tokens one by one. Each generated token is embedded and passed back into the model as the next input. Coconut changes this for the reasoning part. Instead of forcing every reasoning step to be a visible word token, it lets the model think for a few steps in its hidden continuous space.

In my implementation, I used special markers to define the latent reasoning area. The answer starts with <bot>, then the model uses <latent> positions for hidden thinking, and then <eot> marks the end of the latent part. During those latent steps, the model's last hidden state is fed back as the next input embedding directly. This means the model is not choosing a normal vocabulary token during the latent steps. 

Question: ...
Answer: <bot> <latent> <latent> <eot> final answer text

The main idea is that the hidden state can carry reasoning information without converting every intermediate thought into text. After the latent steps are finished, the model switches back to normal language generation and writes the final answer. In this small experiment, I used 2 latent thoughts before decoding the answer.

The curriculum idea is also important. Instead of replacing all reasoning with latent thoughts at once, Coconut can progressively replace parts of the chain-of-thought with latent states. This helps the model learn the hidden-state feedback mechanism more gradually.

3. Problems / issues I considered and how I handled them

Hidden-state feedback loop

The most important part was the latent feedback loop. A latent step cannot simply be treated like a normal token, because there is no real target word at that position. I handled this by using the last hidden state from the model and feeding it back as the next input embedding for the next latent step.

Sequential latent forward pass

The latent part has to be sequential. Each latent step depends on the hidden state from the previous step, so I could not handle the latent section like one static batch of embeddings. I ran the latent steps in order so the feedback mechanism worked correctly.

Special token and marker handling

I added and used the markers <bot>, <latent>, and <eot>. These markers are important because they tell the model where normal text ends, where latent reasoning starts, and where final answer decoding starts.

Loss masking

The latent positions should not be supervised as normal language tokens. The loss should mainly supervise the remaining text answer, not force the model to predict a vocabulary word for the hidden latent steps. This was important for making the overfit experiment meaningful.

Repeated generation

At first, the generated answer repeated the final sentence several times. The numeric answer was correct, but the output was not clean. I fixed this by adding an end/stop condition in generation and by using the clean extracted answer for the report. The final generated text now stops after the correct answer instead of repeating.

Runtime limitation

I also tried moving toward a fuller curriculum-style run, but my Colab runtime disconnected before that experiment could finish. Because of that, I am not claiming a full GSM8K accuracy result.

4. Single-example overfit result

The required bottom-line result was to show that one training example can be driven to near-zero loss. In my final run, the loss started at 1.862770 and reached a final loss of 5.208463e-06 after 200 training steps.

python final_train.py
